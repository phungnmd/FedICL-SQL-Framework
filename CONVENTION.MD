# FedICL-SQL Research Project Convention

> Conventions for this FedICL-SQL research project. Treat this file as law. The goal is simple: **any number in the paper must be traceable to a config, a commit, and a run.**

---

## 1. Guiding Principles

1. **Reproducibility over speed.** A result you can't reproduce doesn't exist.
2. **Notebooks tell stories; modules do work.** Logic that runs twice belongs in `src/`.
3. **Configs over constants.** No hyperparameter, path, or model name is hardcoded.
4. **One experiment = one config + one commit + one tracked run.**
5. **The paper is generated, not typed.** Tables and figures come from scripts, never copy-pasted numbers.

---

## 2. Repository Structure

FedICL-SQL uses a two-repo layout on purpose:

- This outer repo is the private paper workspace: plans, notes, references, paper assets, and project governance.
- `fedicl-sql/` is its own git repo, gitignored here, and is the standalone code/results evidence trail that can be released at submission.
- Commit code-stage progress in the inner repo. Commit plan, paper, reference, and convention changes in the outer repo.
- Reference PDFs and private notes stay in the outer repo and must never be published.

```
<project-name>/
├── README.md               # what, why, quickstart (< 5 min to first run)
├── CONVENTION.MD            # this file
├── pyproject.toml           # deps + tool config (ruff, pytest)
├── uv.lock                  # locked environment (committed)
├── Makefile                 # canonical entry points
├── .gitignore               # data/, experiments/, wandb/, *.ckpt, .env
│
├── configs/                 # ALL experiment configuration (YAML)
│   ├── base.yaml
│   └── exp/                 # one file per experiment / ablation
│
├── data/                    # NEVER committed
│   ├── raw/                 # immutable, read-only originals
│   ├── interim/             # partially processed
│   └── processed/           # model-ready
│       └── <purpose>/       # CSV files + meta.json provenance
│
├── fedicl_sql/              # the package — flat top-level (NOT src/-layout);
│   │                        # importable, tested, no side effects on import
│   ├── data/                # loading, preprocessing, splits
│   ├── models/              # architectures, wrappers
│   ├── training/            # train loops, schedulers
│   ├── eval/                # metrics, statistical tests
│   ├── federated/           # delta aggregation → M_G
│   ├── retrieval/           # ICL demo retrieval
│   ├── prompts/             # prompt builder
│   ├── runtime/             # result logging + provenance
│   └── utils/               # seeding, logging, io
│
├── scripts/                 # thin CLI entry points only
│   ├── train.py
│   ├── evaluate.py
│   └── make_tables.py       # results → LaTeX tables
│
├── notebooks/               # numbered, narrative, re-runnable
│   ├── 01_data_exploration.ipynb
│   ├── 02_prototype.ipynb
│   └── 99_paper_figures.ipynb
│
├── experiments/             # logs + checkpoints NEVER committed; BUT
│   │                        # results/, RUNS.csv, REGISTRY.md ARE committed
│   │                        # (the paper's evidence trail — see §5)
│   └── <exp-name>/results/<run_id>/   # summary.json + rows.csv (committed)
├── artifacts/               # heavy regenerable outputs (LoRA, M_G), NEVER committed
│
├── tests/                   # at minimum: data pipeline + metrics
└── paper/
    ├── main.tex             # or Overleaf git remote
    ├── figures/             # PDFs exported from 99_paper_figures
    └── tables/              # .tex generated by make_tables.py
```

**Rules:**
- `data/raw/` is **immutable**. Nothing ever modifies it. All transforms write to `interim/` or `processed/`.
- FedICL-SQL experiments read processed CSV, never raw JSON directly. Raw Spider JSON and gold SQL are parsed into CSV first.
- Each `data/processed/<purpose>/` folder includes a `meta.json` with source, generation script, parameters (`alpha`, `k`, seed), and split sizes.
- `src/` never reads from absolute paths. Paths come from config.
- `scripts/` contains no logic — only argument parsing, config loading, and calls into `src/`.
- Reusable code belongs in the shared `fedicl_sql/` package. Experiment-specific code belongs in `experiments/<exp-name>/` and imports the shared package.
- Heavy regenerable outputs such as LoRA adapters and `M_G` checkpoints go under `artifacts/<run_id>/`, not committed results folders.

## 3. Notebooks

| Rule                                                                                     | Rationale                                 |
| ---------------------------------------------------------------------------------------- | ----------------------------------------- |
| Prefix with number: `01_`, `02_`, … `99_paper_figures`                                   | Reading order = story order               |
| Must run top-to-bottom after `Restart & Run All` before commit                           | Hidden state is the #1 notebook bug       |
| No function > 10 lines defined in a notebook                                             | Move it to `src/`, import it back         |
| Code used by ≥ 2 notebooks → promote to `src/` immediately                               | Single source of truth                    |
| Pair with `.py` via **jupytext**                                                         | Reviewable diffs in git                   |
| `99_paper_figures.ipynb` reads only from `experiments/`, writes only to `paper/figures/` | Figures are derived artifacts, end to end |

Notebooks are for: exploration, sanity checks, prototypes, and paper figures. Nothing else.

## 4. Configuration

- Format: **JSON run matrices** at `configs/<experiment>/<stage>[_<variant>].json` — one file per planned run definition (see `configs/README.md` for the schema). One config per experiment reported in the paper, named after the table/ablation it supports.
- Every config carries: `experiment`, `stage`, `seeds` (headline lists ≥3), the exact `args` for `run.py` (data paths/splits, model spec, training hyperparameters, eval settings), `paper_artifact`, and a free-text `notes` field.
- New planned run → write the config file FIRST, then execute it. The executed `vars(args)` is mirrored into each run's `summary.json`; diff intended vs executed when a result looks odd.
- **Banned:** hardcoded learning rates, magic paths, `if dataset == "spider"` branches scattered in code.

## 5. Experiments & Tracking

Every run that could appear in the paper:

1. Starts from a **clean git state** (no uncommitted changes; `save_results` stamps the commit `git_sha`).
2. Is launched through `experiments/<exp-name>/run.py` (or `scripts/`) with an explicit JSON config.
3. Persists through `save_results` (full executed config, metrics, `git_sha`, hardware). An external tracker (Weights & Biases / MLflow) is **optional**; the `RUNS.csv` ledger + per-run `summary.json` are the **mandatory** provenance store.
4. Writes outputs to `experiments/<exp-name>/results/<run_id>/`, including a final `summary.json`.

**FedICL-SQL provenance contract:**
- Each experiment writes results through `fedicl_sql.runtime.results.save_results` to `experiments/<exp-name>/results/<run_id>/`.
- Results are never stdout-only. Re-runs and additional seeds never overwrite prior runs.
- Each run folder contains `summary.json` and `rows.csv` with stable column names so results concatenate across experiments.
- `summary.json` must include `experiment`, `model`, `stage`, `seed`, `time_average`, and `gpu_vram`; `save_results` enforces these fields.
- `run_date`, `git_sha`, and `run_id` are auto-stamped. Include the executed config plus experiment metrics such as EX, EM, communication cost, alpha, k, and dataset where applicable.
- `experiments/RUNS.csv` is the append-only run ledger. It carries `device` (`cuda` | `mps` | `cpu`) per run — **never compare numbers across devices in one table** (Apple-Silicon fp16 ≠ CUDA 4-bit QLoRA); keep one compute stack per comparison set.
- `experiments/REGISTRY.md` maps paper artifact -> experiment -> research question -> status, and holds the **cost tally** (external $ not auto-captured). Add a registry row before starting a new experiment.

**Working notes (`paper/notes/`) — three layers by change-frequency:**

- **Spec (rarely changes):** `detailed_plan.md` (what to build) + `fig1_architecture.md` (the mechanism). Reference — read occasionally; do **not** track day-to-day progress inside them.
- **Daily driver:** `todo.md` ▶️ Runbook (the linear step checklist, tick boxes) + `LAB_LOG.md` (dated, **append newest on top**: did / got / next / Q). Write the log every session; read its top entry to resume — never re-parse the spec.
- **Numbers:** `RUNS.csv` (auto) + per-run `summary.json`; headline + the *story* mirrored into `LAB_LOG.md`. Do **not** hand-maintain a parallel results table — that drifts.


**Compute tiers:** Mac M4 Pro (MPS, fp16) = dev + all `$0` no-teacher baselines + PoC; Colab Pro GPU = faster/heavier PoC + 4-bit QLoRA; paid A100-class = Stage-B headline only; teacher 72B = paid API regardless. bitsandbytes 4-bit is CUDA-only.

**Stage discipline:**
- `smoke`: mechanical check that the pipeline runs.
- `poc`: Stage-A free-tier evidence. It gates Stage B but is never cited in the paper.
- `headline`: Stage-B outline-model evidence over at least 3 seeds. Only headline runs are paper-citable.
- Analysis scripts must filter `stage == "headline"` for paper tables and figures.

**Statistical hygiene:**
- Report **mean ± std over ≥ 3 seeds** (5 preferred). Single-seed numbers are placeholders, never paper numbers.
- Fix seeds via one utility: `src/utils/seeding.py` (Python, NumPy, framework, dataloader workers).
- Compare against baselines run **by you, in this repo**, under identical preprocessing — not numbers copied from other papers, unless explicitly marked `†` (reported) in the table.

**Sanity checks before any sweep:**
- Model can overfit a tiny subset (≈ 10 samples). If not, there's a bug.
- Eval metric on a trivial baseline (majority class / random) matches expectation.

## 6. Git Conventions

- **Branches:** `main` (always runs), `exp/<slug>` for risky experiment code, `paper/<venue>` for submission freezes.
- **Commits:** Conventional-style prefixes — `feat:`, `fix:`, `exp:`, `data:`, `paper:`, `refactor:`. An `exp:` commit pins the exact code for a set of runs.
- **Tags:** tag every submission: `v1.0-submitted-<venue>`, `v1.1-camera-ready`. Reviewers' "rerun with X" requests land months later; the tag is your time machine.
- **Never commit:** data, checkpoints, `wandb/`, API keys, `experiments/`. Use `.env` (gitignored) + `.env.example` (committed) for secrets.

## 7. Figures & Tables

- Figures exported as **vector PDF** (not PNG) from `99_paper_figures.ipynb` into `paper/figures/`.
- One shared matplotlib style file (`src/utils/plotting.py`): consistent fonts (match paper body, ~8–9 pt at final size), colorblind-safe palette, no chartjunk.
- Every figure regenerable by one command: `make figures`.
- Tables generated by `scripts/make_tables.py` → `paper/tables/*.tex`, included via `\input{}`. **Typing numbers into LaTeX by hand is forbidden** — it is how wrong numbers get published.
- FedICL-SQL paper tables and figures are generated by `analysis/make_*.py` into `paper/{tables,figures}/`; mean, standard deviation, and significance testing belong there, not inside experiment scripts.
- Bold the best result programmatically, report std in `\small` or as `±`, mark externally-reported numbers with `†`.

---

## 8. Code Quality

- **Formatter/linter:** `ruff` (format + lint), enforced via `pre-commit`.
- **Types:** type hints on all public functions in `src/`; `mypy` optional but encouraged.
- **Tests (minimum viable for research):**
  - Data pipeline: shapes, no leakage between splits, deterministic given seed.
  - Metrics: hand-computed values on toy inputs match implementation.
  - One smoke test: training runs 2 steps end-to-end on dummy data.
- Docstrings: one-line summary minimum; full docstring for anything non-obvious.

---

## 9. Paper Workflow

1. **Related work first** — maintain `paper/notes/related_work.md` with one paragraph per paper: claim, method, gap. Managed alongside Zotero.
2. **Claims drive experiments.** Before running a sweep, write the sentence in the paper that the result will support. If you can't write the sentence, don't run the sweep.
3. **Results section is a contract:** every claim cites a table/figure; every table/figure maps to configs in `configs/exp/`.
4. Writing lives on **Overleaf with git sync** (or `paper/` in-repo). Freeze a branch at submission.

---

## 10. Reproducibility Checklist (pre-submission)

- [ ] `uv.lock` / pinned environment committed; Python version stated in README
- [ ] README contains exact commands to reproduce every table and figure
- [ ] All paper numbers traceable: config file → commit hash → tracked run
- [ ] ≥ 3 seeds, mean ± std reported
- [ ] Hardware and runtime documented (GPU/Apple Silicon, memory, wall-clock)
- [ ] Dataset versions, licenses, and download instructions documented
- [ ] `make figures && make tables` regenerates all paper assets from scratch
- [ ] Repo runs on a clean clone in a fresh environment (test it — actually do it)
- [ ] Anonymized fork prepared if venue is double-blind

## 12. Makefile Targets (canonical interface)

```makefile
setup:      ## create env, install deps, install pre-commit
data:       ## download + preprocess data
train:      ## train with CONFIG=configs/exp/<name>.yaml
eval:       ## evaluate a checkpoint
figures:    ## regenerate all paper figures
tables:     ## regenerate all paper tables
test:       ## run test suite
clean:      ## remove caches (never touches data/raw)
```

If a collaborator (or you, in six months) can't reproduce the paper with `make setup data train eval figures tables`, the project isn't done.
